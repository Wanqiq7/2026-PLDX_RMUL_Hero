## 问题1: ins_task.c HAL函数违规调用

### 问题描述

`modules/imu/ins_task.c` 中直接调用了 HAL 层函数，违反了三层架构原则。

```c
// 修改前 - 直接调用HAL函数
HAL_TIM_PWM_Start(&htim10, TIM_CHANNEL_1);
__HAL_TIM_SetCompare(&htim10, TIM_CHANNEL_1, (uint16_t)output);
```

### 架构违规

```
正确依赖方向: BSP → Module → APP
违规依赖:     Module → HAL (跨层调用)
```

### 修复方案

在 BSP 层封装 PWM 控制接口，Module 层通过 BSP 接口操作硬件。

## 问题2: vision_comm 模块反向依赖

### 问题描述

`modules/vision_comm/vision_comm.c` 中包含了 APP 层的头文件：

```c
// 修改前
#include "robot_def.h"  // APP层头文件，违反依赖方向

// 使用了APP层定义的宏
#if VISION_LINK_TYPE == VISION_LINK_VCP
    // ...
#endif
```

### 架构违规

```
正确依赖方向: BSP → Module → APP
违规依赖:     Module → APP (反向依赖)
```

### 修复方案 (方案B: 运行时配置/依赖注入)

1. 在 Module 层定义配置结构体
2. 初始化时由 APP 层传入配置参数
3. 移除对 `robot_def.h` 的依赖

```c
// 修改后 - vision_comm.h
typedef struct {
    uint8_t link_type;  // 通信链路类型
    // ... 其他配置
} VisionCommConfig_s;

void VisionCommInit(const VisionCommConfig_s *config);
```

## 问题3: vision_control 模块反向依赖

### 问题描述

`modules/vision_control/vision_control.c` 中包含了 APP 层的头文件：

```c
// 修改前
#include "robot_def.h"

// 使用了APP层定义的宏
static float ConstrainPitchRad(float pitch_rad) {
    const float pitch_max_rad = PITCH_MAX_ANGLE * DEG_TO_RAD;  // APP层宏
    const float pitch_min_rad = PITCH_MIN_ANGLE * DEG_TO_RAD;  // APP层宏
    // ...
}
```

### 修复方案 (方案B: 运行时配置/依赖注入)

1. 在配置结构体中添加 pitch 限位参数
2. 函数接受参数而非使用编译期宏

```c
// 修改后 - vision_control.h
typedef struct {
    // ... 其他参数
    float pitch_max_angle;  // [deg] Pitch轴最大角度限位
    float pitch_min_angle;  // [deg] Pitch轴最小角度限位
} VisionCtrlParams_s;

// 修改后 - vision_control.c
static float ConstrainPitchRad(float pitch_rad, float pitch_max_deg, float pitch_min_deg);
```

## 问题4: referee 模块反向依赖

### 问题描述

`modules/referee/rm_referee.h` 和相关文件中包含了 APP 层的头文件：

```c
// 修改前 - rm_referee.h
#include "robot_def.h"

// 使用了APP层定义的枚举
typedef struct {
    chassis_mode_e chassis_mode;  // APP层枚举
    gimbal_mode_e gimbal_mode;    // APP层枚举
    // ...
} Referee_Interactive_info_t;
```

### 修复方案 (方案B: Module层定义UI专用枚举)

在 Module 层定义 UI 专用的枚举类型，与 APP 层解耦：

```c
// 修改后 - rm_referee.h
typedef enum {

    UI_CHASSIS_ROTATE,
    UI_CHASSIS_NO_FOLLOW,

typedef enum {
    UI_GIMBAL_ZERO_FORCE = 0,
    UI_GIMBAL_FREE_MODE,
    UI_GIMBAL_GYRO_MODE,
} ui_gimbal_mode_e;

// ... 其他UI专用枚举
```

## 问题5: PID控制器命名规范不一致

### 问题描述

`modules/algorithm/controller.h` 中 PID 结构体成员使用大写命名：

```c
// 修改前
typedef struct {
    float Kp;  // 应为小写
    float Ki;  // 应为小写
    float Kd;  // 应为小写
    // ...
} PID_Init_Config_s;
```

### 编码规范

根据项目编码规范，变量应使用小写下划线命名法。

### 修复方案

将所有 PID 参数命名改为小写：

```c
// 修改后
typedef struct {
    float kp;
    float ki;
    float kd;
    // ...
} PID_Init_Config_s;
```

### 影响范围

需要同步修改以下文件中的 PID 参数引用：
- `modules/algorithm/controller.c`
- `application/chassis/chassis.c`
- `application/gimbal/gimbal.c`
- `application/shoot/shoot.c`
- `modules/imu/ins_task.c`

## 问题6: 消息中心函数命名规范不一致

### 问题描述

`modules/message_center/message_center.h` 中函数命名不符合动宾短语规范：

```c
// 修改前
Publisher_t *PubRegister(char *name, uint8_t data_len);
Subscriber_t *SubRegister(char *name, uint8_t data_len);
```

### 编码规范

函数命名应使用动宾短语，首字母大写。

### 修复方案

```c
// 修改后
Publisher_t *RegisterPublisher(char *name, uint8_t data_len);
Subscriber_t *RegisterSubscriber(char *name, uint8_t data_len);
```

### 影响范围

需要同步修改以下文件中的函数调用：
- `modules/message_center/message_center.c`
- `application/cmd/robot_cmd.c`
- `application/gimbal/gimbal.c`
- `application/shoot/shoot.c`
- `application/chassis/chassis.c`
- `application/vision/vision.c`
- `application/sysid/sysid_task.c`
